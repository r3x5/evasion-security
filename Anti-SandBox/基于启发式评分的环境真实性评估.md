## 程序整体架构与判定思想

##### 整体结构可以概括为三层：
#### 第一层：离散检查项

每个函数负责验证一个独立维度，例如：

- 时间行为是否正常
- 文件映射与文件缓冲一致性是否正常
- 程序目录是否丰富
- 当前进程集合是否像真实桌面
- 用户目录是否存在近期修改
- 浏览器历史数据库是否存在且足够大
- PowerShell 历史是否存在

这些检查项都是弱信号。单独看，没有任何一个信号可以可靠证明真机或沙箱

#### 第二层：加权聚合

代码没有把某一项当成绝对条件，而是引入了 `CheckItem` 结构体：

```C
typedef struct {  
    const char* name;  
    CheckFunc func;  
    int weight;  
} CheckItem;
```

每个检查项被赋予权重。这样做的意义是：

- 真实机器并不一定满足所有特征，沙箱也可能伪造部分特征。
- 单个检测项被绕过，不会立刻导致整体判定失效。
- 比如“最近有文件改动”通常比“PowerShell 历史存在”更能说明用户活动密度，因此权重更高。

#### 第三层：阈值决策

最终通过总分百分比与阈值比较：

```C
if (score >= 60) {
    printf("[+] Real environment\n");
} else {
    printf("[!] Suspicious environment\n");
}
```

这里采用的是一种启发式，容错式判定，而不是确定性证明


## 部分模块设计原理

#### `CheckBehaviorConsistency()`：检查文件映射与文件系统行为一致性

该函数通过如下流程测试底层文件行为的一致性：

1. 在临时目录创建一个新文件
2. 对该文件建立文件映射
3. 将映射视图中的内容写成特定字节模式
4. 将首字节改成 `'B'`
5. 刷新视图与文件缓冲
6. 从文件句柄重新读取首字节
7. 验证读回来的内容是否仍是 `'B'`

在正常 Windows 主机上：

- `CreateFileA` 打开文件对象
- `CreateFileMappingA` 创建基于该文件的 section object
- `MapViewOfFile` 将 section 映射到进程地址空间
- 对映射页的修改会进入缓存管理路径
- `FlushViewOfFile` / `FlushFileBuffers` 应促使映射与文件内容保持一致
- 后续 `ReadFile` 读到的应是更新后的字节

#### `DelayExecution()`：检查时间推进是否真实

该函数随机生成 4~7 秒的睡眠时间，记录睡眠前后的 tick 值，并判断实际经过时间是否至少达到目标延迟的 80%。之所以是80%，不是要求绝对精确，而是允许一定误差。因为Windows调度存在天然误差，且系统负载会导致唤醒稍晚。


#### `CheckFolders()`：检查常见软件安装痕迹

函数遍历 `C:\Program Files\` 下的一组常见软件目录，例如：

- Adobe
- Google
- Java
- Steam
- Office
- NVIDIA
- Blender
- WinRAR
- Notepad++

命中数量达到 3 个即判定通过，因为真实用户终端通常具备如下特征：

- 安装过浏览器、办公软件和游戏平台
- 软件目录具有一定多样性
- 文件系统状态呈现出长期积累而非一次性构造

而很多分析沙箱为了追求启动速度和快照可重复性，会刻意保持系统干净轻量


#### `CheckProcesses()`：检查当前活动进程集合是否像真实桌面环境

该函数通过 `CreateToolhelp32Snapshot` 枚举系统进程，并匹配一批常见应用进程，命中两个以上即通过

```C
int hits = 0;
int matched[100];
ZeroMemory(matched, sizeof(matched));

if (Process32FirstW(snap, &pe)) {
	do {
		for (int i = 0; targets[i] != NULL; i++) {
			if (matched[i]) continue;
			if (_wcsicmp(pe.szExeFile, targets[i]) == 0) {
				matched[i] = 1;
				hits++;
				wprintf(L"[+]     Found Process: %ls\n", pe.szExeFile);
				break;
			}
		}
	} while (Process32NextW(snap, &pe));
}
```

由于有些程序会开多个子进程，所以代码使用 `matched[i]` 防止同一目标重复计数。


#### `IsRecentChange()` `ScanDirectory()` `CheckRecentChanges()`：检查文件系统是否存在近期活跃痕迹

该部分代码做了三件事，只要检测到近期变化的文件即认为存在用户活动痕迹：

1. 枚举 `C:\Users\*`
2. 递归扫描各用户目录
3. 判断文件的创建时间或最后写入时间是否在近 7 天内

这里使用 [WIN32_FIND_DATAA](https://learn.microsoft.com/en-us/windows/win32/api/minwinbase/ns-minwinbase-win32_find_dataa) 检测 `ftLastWriteTime`, `ftCreationTime` ，因为沙箱的特征通常是，镜像固定，每次运行从同一快照恢复，且用户目录内容稀少


#### `CheckBrowserSessions()`：检查浏览器历史数据库是否存在并具有一定体量

浏览器是现代用户终端最稳定最普遍的活动来源之一。相比单纯检查浏览器安装目录，检查历史数据库更有意义。

而文件是否存在只能说明可能被创建过，一个极小的历史数据库可能只是首次启动产生， 
一个较大的历史库更像长期使用结果。所以我增加了判断

```C
if (hFind != INVALID_HANDLE_VALUE) {
	ULONGLONG size = ((ULONGLONG)fd.nFileSizeHigh << 32) | fd.nFileSizeLow;

	if (size > 100 * 1024) {
		score += 2;
	}
	else {
		score += 1;
	}
	FindClose(hFind);
}
```

计算文件大小的方式为将高32位左移32位，再与低32位进行按位或操作，得到64位总字节数


#### `CheckPSHistory()`：检查 PowerShell 历史记录

函数检查：

```PowerShell
AppData\Roaming\Microsoft\Windows\PowerShell\PSReadLine\ConsoleHost_history.txt
```

如果存在且大于 5KB，则判定为通过。

与进程检测不同，这是一种更偏向特定人群的信号，因此它不该占用太高权重。
